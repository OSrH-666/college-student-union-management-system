## 一、项目概述

### 1. 项目背景
当前高校社团、学生会组织数量众多，传统管理依赖纸质表单、群消息与人工统计，存在信息传递不及时、数据易丢失、流程繁琐等问题。为提升社团与学生会管理效率、规范运营流程、优化成员体验，本项目拟构建覆盖“组织创建—审核—运营—活动—总结”全流程的一体化数字平台。该平台将作为《信息系统实践》课程设计，重点实践数据库设计、SQL 编程及前后端分离开发。

### 2. 项目目标
- 实现学生、社团/学生会负责人、系统管理员三级角色协同工作平台。
- 完成组织创建、成员审核、活动发布、在线报名、多方式签到及活动总结的全流程闭环。
- 引入 RBAC 权限控制，保障系统安全与数据隔离。
- 实现活动报名实时限额及等待队列机制，提升用户体验。
- 出具规范的数据库设计文档（ER 图、关系模式、建表脚本），满足课程设计要求。

### 3. 开发环境与条件
- **开发工具**：IntelliJ IDEA / VS Code，Navicat，Git，Postman
- **数据库**：MySQL 8.0（InnoDB 引擎，字符集 utf8mb4）
- **后端**：Java 8/11 + Spring Boot 2.x，MyBatis-Plus，Spring Security + JWT
- **前端**：Vue3 + Vite，Element Plus，Axios
- **运行环境**：本地开发机，可选 Docker 容器化部署

## 二、需求分析

### 1. 目标用户与使用场景
- **学生**：浏览社团/学生会信息、申请加入、查看活动、在线报名、取消报名、参与签到、查看个人参与记录。
- **社团/学生会负责人**：管理组织基本信息、审核成员申请、发布活动、设置报名限额、查看报名名单、执行多方式签到、撰写活动总结。
- **系统管理员**：管理用户账号、分配角色权限、审核组织创建、查看统计数据、系统配置。

### 2. 核心功能清单
- 组织管理：社团/学生会创建与审核、信息维护、成员申请与审批、角色分配
- 活动管理：活动发布、时间地点设定、报名限额、活动详情展示
- 报名与等待队列：实时显示剩余名额，满额自动进入等待队列；取消报名时按顺序递补
- 多方式签到：支持扫码签到、地理位置签到、管理员手动签到，记录签到方式与时间
- 活动总结：负责人撰写总结，支持附件上传，成员可查看
- 权限控制：基于 RBAC，管理员可分配角色；菜单和接口按角色隔离

### 3. 非功能需求
- **性能要求**：页面响应时间不超过 2 秒，支持至少 50 人并发报名时的等待队列正确处理。
- **安全要求**：密码加密存储，JWT 令牌过期机制，跨角色数据隔离，防止越权操作。
- **兼容性要求**：前端兼容 Chrome/Firefox/Edge 最新版本；后端 RESTful 接口遵循规范，便于联调。

## 三、技术路线

### 1. 总体架构
采用前后端分离架构：Vue3 前端通过 HTTP 请求调用 Spring Boot 提供的 RESTful API；后端按 Controller – Service – Mapper 分层设计，通过 MyBatis 操作 MySQL 数据库；引入 Spring Security 结合 JWT 实现认证与 RBAC。

### 2. 主要技术
- 后端：Spring Boot 2.x，MyBatis-Plus，Spring Security，JWT
- 前端：Vue3 (Composition API) + Vite，Element Plus，Axios，Vue Router
- 数据库：MySQL 8.0，使用 InnoDB 引擎，设计时遵循第三范式，关键事务用悲观锁或乐观锁保证一致性
- 协作：Git + 在线仓库（如GitHub），前后端 API 文档用 Swagger 辅助定义

### 3. 数据设计初步考虑
- **主要实体或对象**：用户、角色、权限、组织（社团/学生会）、组织成员、活动、报名记录、签到记录、活动总结。
- **关键数据表及关系**：
  - `user`（用户）与 `role`（角色）多对多，通过 `user_role` 关联，实现 RBAC。
  - `club`（组织）与 `user` 通过 `club_member` 关联，同时记录成员状态与组织内角色。
  - `activity`（活动）属于 `club`，一个组织可有多个活动。
  - `registration`（报名）关联活动与用户，包含状态字段以支持等待队列；活动限额满时新报名置为等待状态。
  - `check_in`（签到）与 `registration` 一对一关联，记录签到方式与时间。
  - `summary`（活动总结）与 `activity` 一对一关联，保存总结内容与附件路径。

> 后续将绘制 ER 图并细化各表字段、约束与索引，确保满足业务需求与并发场景。
